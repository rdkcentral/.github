# Contributing to RDK Central 🤝

Thank you for your interest in contributing to RDK Central! We welcome contributions from developers, organizations, and community members worldwide.

## 🚀 Quick Start

1. **Fork** the repository you want to contribute to
2. **Clone** your fork locally
3. **Create** a feature branch for your changes
4. **Make** your changes following our guidelines
5. **Test** your changes thoroughly
6. **Submit** a pull request

## 📋 Before You Start

### Contributor License Agreement (CLA)
Before RDK accepts your code into the project, you must sign the RDK Contributor License Agreement (CLA). This ensures that contributions can be freely used by the community while protecting both contributors and users.

- **Individual CLA**: For personal contributions
- **Corporate CLA**: For contributions made on behalf of an organization

### Code of Conduct
All contributors must follow our [Code of Conduct](CODE_OF_CONDUCT.md). We're committed to providing a welcoming and inclusive environment for everyone.

## 🛠️ Development Setup

### Prerequisites
- Git
- Appropriate build tools for your target RDK component
- Development environment (see component-specific documentation)

### Setting Up Your Environment
1. Fork the repository on GitHub
2. Clone your fork:
   ```bash
   git clone https://github.com/yourusername/repository-name.git
   cd repository-name
   ```
3. Add upstream remote:
   ```bash
   git remote add upstream https://github.com/rdkcentral/repository-name.git
   ```
4. Install dependencies (component-specific)

## 📝 Contribution Types

### 🐛 Bug Fixes
- Use the bug report template when reporting issues
- Include reproduction steps and environment details
- Reference the issue number in your pull request
- Add regression tests when possible

### ✨ New Features
- Start with a feature request to discuss the proposal
- Break large features into smaller, reviewable changes
- Include comprehensive tests and documentation
- Update relevant documentation

### 📚 Documentation
- Fix typos, clarify explanations, add examples
- Update API documentation for code changes
- Translate content to other languages
- Create tutorials and guides

### 🧪 Testing
- Add unit tests for new functionality
- Improve test coverage for existing code
- Add integration tests
- Performance testing and benchmarking

## 📐 Coding Standards

### Code Style
- Follow existing code style in the repository
- Use consistent naming conventions
- Add comments for complex logic
- Remove unused imports and variables

### Commit Messages
Follow conventional commit format:
```
type(scope): brief description

Longer explanation if needed

Fixes #123
```

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`

### Pull Request Guidelines
- Use the pull request template
- Keep PRs focused and reasonably sized
- Include tests for new functionality
- Update documentation as needed
- Ensure all CI checks pass

## 🔍 Review Process

### What to Expect
1. **Automated Checks**: CI/CD pipelines run automatically
2. **Code Review**: Maintainers and community review your changes
3. **Feedback**: Address review comments and suggestions
4. **Approval**: Once approved, your PR will be merged

### Review Criteria
- Code quality and adherence to standards
- Test coverage and quality
- Documentation completeness
- Performance impact
- Security considerations
- Backward compatibility

## 🏗️ Component-Specific Guidelines

### RDK-V (Video)
- Focus on performance and memory efficiency
- Consider multi-platform compatibility
- Test with various media formats and codecs

### RDK-B (Broadband)
- Ensure network security best practices
- Test with different network configurations
- Consider performance under load

### RDK-C (Camera/IoT)
- Focus on power efficiency
- Test device compatibility
- Consider privacy and security implications

### Thunder Framework
- Maintain plugin architecture principles
- Ensure thread safety
- Test plugin interactions

## 🚨 Security Contributions

For security-related contributions:
- Follow our [Security Policy](SECURITY.md)
- Do not include security issues in public PRs
- Contact security@rdkcentral.com for sensitive matters
- Use responsible disclosure practices

## 📊 Performance Contributions

When contributing performance improvements:
- Include benchmarks and measurements
- Test on relevant hardware platforms
- Document performance impact
- Consider memory usage implications

## 🌍 Internationalization

Help make RDK accessible globally:
- Translate documentation and user interfaces
- Ensure proper character encoding support
- Test with international configurations
- Follow localization best practices
